This is a beginner's guide to design and build accessible web application. More specifically, this is a guide to help you build websites in a way that people with disabilities (such as visual, physical or neurological) are be able to understand, interact and contribute just as easily as a normal person can.

It might be mildly surprising that you can browse this page with your eyes closed right now! Applications, called screen readers are meant to do just that. ChromeVox is one such (free) plugin for Google Chrome browser. Screen readers use a combination of keyboard and text to speech to aid visually impaired users browse the web. So imagine you type in the url of a page and when the page is loaded the screen reader starts reading out the content and when you press Tab it reads out the first actionable link on the page and when you hit Enter it opens that link. Of course browsing a page can be way more complex than that and unfortunately most of the websites are not designed to be compatible with screen readers.. But if you design your website in a way that will help the screen readers do their job then you can enable a visually impaired user do exactly as much as a normal user can.

Tools that help individuals with disabilities to browse the web are termed as Assistive Technology (AT).

ATs include tools like
  • Wands and sticks worn on the head, held in the mouth or strapped to the chin and used to press keys on the keyboard
  • Screen readers are used to verbalize, or "speak," everything on the screen including text, graphics, control buttons, and menus into a computerized voice that is spoken aloud
  • Screen magnifiers work like a magnifying glass for the computer by enlarging a portion of the screen which can increase legibility and make it easier to see items on the computer

Much like how W3C defined HTML5 to standardise HTML markup, Web Content Accessibility Guidelines (WCAG) defined guidelines to make web applications more accessible. Which means that a developer who want to make his website usable via ATs have to follow these guidelines. The aim of this guide is not to decrypt the plethora of WCAG guidelines but to inspire developers with the philosophy behind it. In order to program websites that are compliant with WCAG you will needs to make a paradigm shift, just like jQuery to Angularjs or non-functional to functional programing. The following section is designed to help you make that paradigm shift with a list of live examples and code snippets which will help you in looking through an AT's eye.
